## PART D — OTP-based signup/login (no external SMS)

Important constraint: You requested **no external SMS providers**. Clarify tradeoff:

* To deliver OTP to a phone/phone number you normally need an SMS gateway (Twilio, Nexmo, provider). Without any external transport, server cannot reach the user's phone number.
* For email OTP, you need SMTP or an external email provider (or run your own SMTP server).
* For development and for local 100% internal testing, implement an **internal OTP service** that **generates and stores OTPs** in DB and returns a masked “delivery hint” to the client. For production, recommend configuring a real SMS/email provider.

Implement the following **secure internal passwordless flow** (works for dev & can be production-ready if you plug a transport later):

Backend:

* `backend/models/tb_otp.py` — store OTP doc:

  ```json
  {
    "_id": ObjectId,
    "destination": "email|phone|user:id if device", 
    "code": "6-digit",
    "expires_at": datetime,
    "used": False,
    "deliver_method": "mock" | "email" | "sms"
  }
  ```
* Routes:

  * `POST /api/auth/otp/send` with `{ destination: "<email or phone>", type: "login" | "signup" }`

    * server: generate random 6-digit, save TBOTP with expires (5 minutes), deliver:

      * If `EMAIL_ENABLED`/`SMS_ENABLED` env true, attempt to send via provider (if configured).
      * Else: return a masked delivery hint in responses; do NOT return plaintext OTPs in API responses. In production, always send OTPs via a configured provider.
    * Rate limit: ensure max 5 attempts per hour per destination.
  * `POST /api/auth/otp/verify` with `{ destination, code, device_id }`

    * Validate code not expired and not used.
    * Create user (signup) or log in user (login). Create session with a `device_id` (UUID).
    * Issue JWT. Store session doc with `user_id`, `device_id`, `jwt_jti`, `created_at`.
    * If user already has an active session for another device and you enforce single-device, revoke previous session (see below).
* `backend/services/token_service.py`:

  * On login, implement single-device enforcement: if a user logs in from `device_B`, mark the previous session's JWT `is_revoked` True and send a socket notification to that device to logout (if connected). Blacklist previous token (store jti in `token_blacklist`).
* Security:

  * OTP codes must be hashed if you want extra security (store HMAC), but storing plaintext OTP in dev is ok. Add attempt count and expiration.

Frontend:

* Login screen: offer Phone/Email OTP flow in addition to password.
* OTP UI: send request to `/api/auth/otp/send`, then show OTP input UI. For dev, show the dev_otp if present (use env flag).
* After verify: store JWT in secure storage (consider httpOnly cookies in future), call `authStore.login()`.

Production recommendation (explain clearly in the implementation notes):

* To guarantee 100% reliable OTP delivery in production, **configure an SMS/email provider** and set `SMS_ENABLED` or `EMAIL_ENABLED` in `.env` pointing to provider credentials. Provide step-by-step integration notes for Twilio/Razorpay/SendGrid.
